### Q1：工作了两年又回去读研，当时为什么做这个决定？

#### 面试官真正想问什么

表面问"为什么读研"，实际在考察三件事：

1. **决策能力**：你的人生重大选择有没有经过理性分析？还是凭感觉？
2. **自驱力**：你是被工作淘汰了才回学校，还是主动追求更高的目标？
3. **读研的投入产出比**：这两年你学到了什么？值不值？

面试官的潜台词是："招你进来，你会不会干两年又跑了？"

#### 强候选人如何回答

采用 **"问题→瓶颈→决策→收获"** 四段式结构：

```
【问题】在工作中负责了一个对话系统的项目，发现自己对语言模型的理解
       停留在"调包"层面，不知道模型为什么有时候好有时候差。

【瓶颈】想深入优化的时候遇到了天花板——看论文看不懂推导，
       调参数靠猜，跟研究员沟通时抓不住重点。意识到自己缺的不是经验，
       而是第一性原理的理解。

【决策】确定了方向（NLP/大模型），选好了导师（该方向的活跃研究者），
       带着工作中攒下的具体问题去读，效率远高于本科直升。

【收获】两年下来，不仅补了理论短板，还发了一篇相关的论文。
       现在做技术决策时，能从原理出发推导，而不是凭经验猜。
```

#### 通俗故事：老木匠回学校

> 有个木匠，在装修队干了两年，手艺很好，柜子打得又快又漂亮。有一天老板接了个大活——设计一个博物馆的展厅，要画图纸、算承重、选木材。
>
> 木匠拿着铅笔坐了三天，发现自己根本不会画设计图。他只会按图纸做，不会自己画图纸。
>
> 于是他决定回学校学建筑设计。同学笑他："我们都比你小五岁，你回来干嘛？"
>
> 两年后他毕业了。再找工作的时候，面试官问他为什么回学校，他说：
>
> "我以前只会做柜子，现在我能在做柜子之前告诉你：为什么这个柜子应该用橡木而不是松木，为什么柜门应该朝左开而不是朝右开，以及——这个柜子到底应不应该做。"
>
> 面试官当场给了 Offer。

#### 面试中常见的踩坑回答

| 踩坑回答 | 为什么踩坑 |
|---------|-----------|
| "工作太累了想休息一下" | 显得没有职业追求 |
| "家里让我读的" | 没有自主决策能力 |
| "本科生不好找工作" | 消极逃避，而不是主动追求 |
| "想拿个文凭" | 目标太功利，没有内在驱动 |

### Q2：最近有了解哪些行业资讯/论文？

#### 面试官真正想问什么

这题的目的不是考你记忆力，而是判断**你跟不跟得上这个领域的节奏**。

AI/NLP 领域的变化速度是按"周"算的——一个你两个月没关注的方向，可能整个范式都变了。面试官想知道：招你进来，你需不需要从头补课？

#### 强候选人如何回答

**不要报菜名。** 说 2-3 个你真正读过、有自己判断的方向，结构是：

```
【方向】最近在关注 XX 方向的进展
【具体内容】读了 YY 团队的 ZZ 论文/博客
【我的思考】它解决的核心问题是什么，方法有什么巧妙之处
【和我的关联】这个思路对我正在做的 XX 有什么启发，
            但它的 ZZ 假设在我们的场景下可能不成立，因为...
```

举例：

> "最近在关注推理加速方向。读了 DeepSeek 关于 Multi-head Latent Attention 的论文，核心思路是把 KV Cache 压缩到一个低维潜在空间，大幅减少了推理时的显存占用。
>
> 对我做部署优化的启发是：不一定非要换更小的模型，可以从注意力机制本身去优化。但 MLA 需要对训练架构做改动，我们目前用的是开源模型，直接应用有门槛。
>
> 所以我也在关注不需要重新训练的加速方案，比如 FlashAttention-3 和量化感知的 KV Cache 裁剪。"

#### 通俗故事：两个厨师的差距

> 两个厨师去应聘。面试官问："最近餐饮界有什么新动向？"
>
> **厨师 A**："有分子料理、低温慢煮、发酵料理、可持续食材..."（报菜名）
> 面试官："那低温慢煮你试过吗？"
> 厨师 A："呃...看过视频。"
>
> **厨师 B**："我最近在尝试低温慢煮。核心原理是把食材真空密封后在恒定低温水中长时间加热，60 度的牛肉煮 48 小时，出来的肉嫩得完全不一样。
>
> 我自己试了三次：第一次温度太高失败了，第二次时间不够，第三次成功了。但我发现对于高频出餐的餐厅来说，48 小时太慢，不太实用。所以我在研究怎么用压力锅模拟类似效果，缩短时间。"
>
> 厨师 B 被录取了。因为他不是"听说的"，他是"做过并思考过的"。

### Q3：你平时用什么 AI 编程工具？主要用来做什么？DeepSeek 才出了一个多月，那你之前用什么？

#### 面试官真正想问什么

1. **效率意识**：你会主动找工具提升自己吗？还是给什么用什么？
2. **对工具的认知深度**：你是"浅尝辄止"还是"用透用精"？
3. **是否有判断力**：DeepSeek 为什么好？你真的对比过，还是跟风？

#### 强候选人的回答框架

```
【工具矩阵】我在不同场景用不同的工具：
  - 写新功能：用 XXX，因为它在代码生成方面最强
  - 阅读陌生代码：用 YYY，因为它的代码解释最清晰
  - Debug：用 ZZZ，因为它能理解堆栈信息
  - 写文档/测试：用 WWW，因为这些任务不需要强推理

【切换原因】从 Copilot 切到 DeepSeek 的原因是：
  - 中文场景下代码注释和变量命名的理解更好
  - 长上下文的处理更稳定（能理解整个项目的结构）
  - 但 XX 场景我还是用老工具，因为 DeepSeek 在 XX 上不如它

【对 AI 编程的思考】AI 编程工具改变了我的工作方式但不改变我的责任。
  代码是 AI 写的，但架构和正确性是我负责的。
  AI 写的东西我一定会 Review，就像 Review 一个实习生的代码。
```

#### 通俗故事：换了新工具的老师傅

> 一个修车师傅去面试，面试官看他工具箱里有一把新扳手。
>
> "这把扳手才出来两个月，你以前用什么？"
>
> "以前用那把蓝色的。但这把新的是棘轮扳手，拧狭窄地方的螺丝不用来回倒手，省一半时间。"
>
> "那你为什么工具箱里还有蓝色的？"
>
> "因为棘轮的扭矩不够大，大螺丝拧不动时还得用蓝的。工具嘛，不是越新越好，是什么活用什么好。我这箱子里每一把都有它的用处。"
>
> 面试官一听就知道：这人是真干活的，不是攒工具的。

## 三、模型选型类（核心战场）

> 这是整场面试中追问最密集的部分。面试官从"为什么选 32B"一路追问到"量化选型"再到"怎么发现问题"，形成了一个完整的技术决策链路。以下逐题展开。

---

### Q4：为什么用千问 2.5 32B？不用 7B 或 72B 的考量是什么？

#### 面试官真正想问什么

这不是一个技术问题，这是一个**决策能力**问题。

面试官不关心你用的是千问还是万问，他关心的是：**你做技术选型的时候，有没有一套方法论？** 这套方法论是否可迁移到其他决策场景？

#### 选型框架：三维权衡模型

任何一个模型选型，本质上是在三个维度之间做 trade-off：

```
           效果 (Quality)
            /\
           /  \
          /    \
         /  最优点 \
        /    ✦     \
       /            \
      /______________\
  成本(Cost)        延迟(Latency)
```

**不会有模型三个维度全占**。你的任务是找到业务的"容忍边界"：

- 效果低于 X 分，用户会走 → 这是**效果底线**
- 延迟超过 Y 毫秒，用户等不了 → 这是**延迟底线**
- 成本超过 Z 万/月，老板不批 → 这是**成本底线**

然后找满足三条底线的模型里效果最好的那个。

#### 32B vs 7B vs 72B 的具体对比

| 维度 | 千问 7B | 千问 32B | 千问 72B |
|------|--------|---------|---------|
| **单卡能跑吗** | 能（甚至消费级显卡） | 能（A100/A10） | 不能（需 2-4 卡） |
| **推理延迟（首 Token）** | ~100ms | ~300ms | ~800ms |
| **推理延迟（生成速度）** | ~80 tok/s | ~40 tok/s | ~20 tok/s |
| **指令遵循能力** | 6/10 | 8/10 | 9/10 |
| **口语化自然度** | 5/10 | 8/10 | 8.5/10 |
| **复杂推理** | 4/10 | 7/10 | 9/10 |
| **月部署成本** | 约 5000 | 约 15000 | 约 50000+ |
| **业务适用度** | 不达标 | ✅ 最优 | 超标（成本） |

#### 决策推导示例

```
Step 1: 确定业务核心需求
  → 口语化对话场景，需要自然的对话体验 + 基本的任务执行

Step 2: 确定三底线
  → 延迟底线：首 Token < 500ms（用户等待极限）
  → 成本底线：月部署成本 < 3万
  → 效果底线：口语化自然度 > 7/10

Step 3: 淘汰不合格选手
  → 72B：首 Token 800ms 超底线，成本 5 万超底线 → 淘汰
  → 7B：口语化自然度 5/10 不达底线 → 淘汰
  → 32B：全部达标，效果最好 → 选定

Step 4: 上线后验证
  → 实际延迟 P99 < 450ms ✓
  → 实际成本 1.5 万/月 ✓
  → 口语化 AB Test 显著正向 ✓
```

#### 通俗故事：选快递员

> 你要招一个人送快递，面试了三个：
>
> **小七（7B）**：骑电动车，跑得飞快（延迟低）。但让他送生鲜冷链，他不会看温度计，冰激凌化成水了还往里送（效果差）。工资 5000。
>
> **大七（72B）**：开冷链大卡车，什么都能送，保温箱、温度计、GPS 全程监控（效果好）。但工资要 5 万，而且车大，小区进不去，每次要打电话让客户出来拿（延迟高）。
>
> **三十（32B）**：开一辆小面包，配了保温箱，生鲜能送，速度还行，小路也能钻。工资 1 万 5。
>
> 你的业务是社区生鲜配送——不是送火箭零件（不需要大七的顶级能力），也不是送不着急的快递（小七的温度控制不够）。你要的就是三十这号人。
>
> 面试官问"为什么选三十"——答案不是"三十最好"，而是**"三十二是我三个底线下最好的选择"**。

### Q5：7B 的性能不应该比 32B 更好吗？（面试官故意的反问）

#### 面试官真正想问什么

这是一个**压力测试**。面试官在故意说一个反常识的命题，看你会不会动摇。

7B 确实更快——延迟更低、吞吐更高。但面试官故意把"速度"说成"性能"，把两个维度混在一起，看你能不能发现这个偷换概念。

#### 关键区分：速度 ≠ 效果

```
速度（Speed）指标：
  - 首 Token 延迟：用户发出请求到看到第一个字的间隔
  - 生成速度：每秒输出多少个 Token
  - 吞吐量（QPS）：每秒处理多少个请求

效果（Quality）指标：
  - 指令遵循准确率：有没有按用户说的做
  - 生成质量：输出的文本好不好、自然不自然
  - 知识准确率：说的对不对
  - 推理能力：能不能处理多步骤的复杂问题
```

7B 在速度维度上完胜，在效果维度上完败。面试官说"性能"时故意不区分这两个维度。

#### 强候选人的回击方式

> 面试官："7B 的性能不应该比 32B 更好吗？"
>
> 强候选人："如果'性能'指的是推理速度，确实 7B 更快。但我们说的'性能'指的是**任务完成的质量**，在这个维度上 7B 不够。
>
> 我们实测过：同样的对话场景下，7B 的口语化回复质量比 32B 低了约 30%。虽然它每秒能吐出更多字，但吐出来的东西用户不满意。
>
> 就像一辆跑车和一辆货车——跑车确实快，但你说的是运货能力。"

#### 通俗故事：学霸 vs 快枪手

> 两个学生考试：
>
> **小七**：做题飞快，半小时交卷，正确率 60%。
> **三十**：做题慢一点，一小时交卷，正确率 90%。
>
> 面试官说："小七不应该更好吗？他半小时就做完了！"
>
> 你说："快是用处不大，关键是要对。用户要的是正确答案，不是快速错误答案。"

### Q6：72B 你说跑不起来成本高，那为什么不用 7B ？

#### 面试官真正想问什么

这题在测试：你淘汰 7B，**是不是真的因为 7B 不够好，还是一开始就冲着 32B 去的**？

面试官要你自证：你把 7B 和 32B 都实际跑了对比，发现 7B 不达标，然后才淘汰了 7B。而不是先入为主选了 32B。

#### 正确回答逻辑

```
1. 我们确实跑了对比实验（不是凭感觉）
2. 7B 在以下具体指标上不达标：
   - 口语化自然的回复占比：7B 只有 60%，32B 有 85%
   - 多轮对话上下文丢失率：7B 在 5 轮后丢失率 25%，32B 只有 8%
   - 复杂指令遗漏率：7B 遗漏率 30%，32B 只有 12%
3. 淘汰 7B 是因为"不及格"，不是因为"不喜欢"
4. 淘汰 72B 是因为"付不起"，不是因为"不好"
5. 32B 是唯一的"又及格又付得起"的选项
```

#### 通俗故事：淘汰赛

> 公司招人，三个人应聘，你给出的结果是：
>
> - **7B（新人）**：笔试 60 分，面试 55 分，薪资 5000 → **不达标，直接淘汰**
> - **32B（中级）**：笔试 85 分，面试 80 分，薪资 15000 → **达标 + 能接受**
> - **72B（资深）**：笔试 95 分，面试 92 分，薪资 50000 → **太贵了，淘汰**
>
> 面试官问："你说 72B 太贵了，那为什么不招 7B 那个便宜的？"
>
> 你答："不是钱的问题。7B 考试没及格，他便宜我也不能要。我们是在及格的人里找性价比最高的，不是在所有人里找最便宜的。"

### Q7：你说那性能是什么性能？

#### 面试官真正想问什么

这个问题说明候选人之前用"性能"这个词太含糊了。面试官在要求你**精确化你的表述**。

技术面试中，"性能"是一个被严重滥用的词。它可以指 10 种不同的东西。面试官逼你明确说出来，考察你的**表达精确度**。

#### "性能"的 10 种含义

| 维度 | 英文 | 含义 | 关注场景 |
|------|------|------|---------|
| 吞吐量 | Throughput | 每秒处理多少请求 | 高并发服务 |
| 延迟 | Latency | 一个请求多久返回 | 实时对话 |
| 准确率 | Accuracy | 输出正确的比例 | 事实性问答 |
| 困惑度 | Perplexity | 模型对文本的"熟悉程度" | 生成流畅度 |
| 召回率 | Recall | 该输出的都输出了吗 | 信息提取 |
| 指令遵循率 | Instruction Following | 有没有听用户的话 | 对话助手 |
| 推理深度 | Reasoning Depth | 能处理多复杂的问题 | 逻辑推理 |
| 生成多样性 | Diversity | 每次回答是否不同 | 创意写作 |
| 鲁棒性 | Robustness | 输入轻微变化是否稳定 | 异常输入处理 |
| 口语化度 | Conversationality | 说话像不像人 | 语音助手 |

下次面试官说"性能"的时候，反问一句："你指的是哪个维度的性能？"——这句话本身就能加分。

## 四、评估指标类（决定胜负）

---

### Q8：口语化能力怎么定义和评测？

#### 面试官真正想问什么

这是整个面试中**最能区分水平**的问题。

初级工程师会用模型，中级工程师会选模型，高级工程师会**定义标准**。面试官的问题本质是："你能不能把你脑子里的模糊概念，变成可量化、可复现、可优化的指标体系？"

这道题决定你的面试结果。

#### 定义框架：从模糊到精确

**第一层：直觉定义（不够）**
> "口语化就是说话更像人，不那么机械。"

**第二层：维度拆解**
口语化能力可以拆成 5 个可度量的子维度：

```
口语化能力
├── 1. 词汇口语度：是否使用口语词汇
│   例："咋了" vs "请问有什么可以帮助您的"
│   度量：口语词汇占比 / 正式词汇占比
│
├── 2. 句式灵活度：句子长度是否有自然波动
│   例：全是 12 字一句的 → 机器味重
│   度量：句长标准差、长短句交替频率
│
├── 3. 语气自然度：是否有语气词、停顿
│   例："嗯...那个..." vs 全是完整句子
│   度量：语气词密度、不完整句比例
│
├── 4. 上下文呼应：是否记得前面说了什么并呼应
│   例："像你刚才说的那个..." vs 每句话独立
│   度量：指代消解的准确率、话题连贯性
│
└── 5. 共情能力：是否体现了对用户状态的理解
│   例："辛苦了"、"听起来你很着急"
    度量：情感标注一致性
```

**第三层：评测方法**

| 方法 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| 人工标注 | 标 1000 条数据，"口语化"打 1-5 分 | 最准确 | 贵、慢 |
| GPT-Eval | 用 GPT-4 当评委打分 | 快、便宜 | GPT 自己也有偏好 |
| 规则指标 | 统计口语词/语气词占比 | 极快、免费 | 粗糙，容易漏判 |
| 用户 AB Test | 让真实用户对比选择 | 最真实 | 最慢、最贵 |
| 困惑度(PPL) | 在口语化语料上算困惑度 | 客观 | 有局限 |

**最佳实践**：离线用 GPT-Eval + 规则做快速筛选，关键节点用人工标注做校准，最终上线用 AB Test 验证。

#### 通俗故事：什么叫"会聊天"？

> 你要从 100 个人里挑一个来做客服。标准是"会聊天"。
>
> 你怎么挑？
>
> **随便挑（第一层）**：看一眼，"这个人笑眯眯的，应该会聊天吧"——这就是凭感觉，不可靠。
>
> **拆开看（第二层）**：你定了 5 条标准：
> 1. 会不会用"呀、嘛、哈"这些词？（词汇口语度）
> 2. 说话长短有没有变化？（句式灵活度）
> 3. 会不会自然地"嗯"和停顿？（语气自然度）
> 4. 记不记得你刚才说了什么？（上下文呼应）
> 5. 有没有表现关心？（共情能力）
>
> **量出来（第三层）**：你给 100 个人同时发一条消息，把回复录下来：
> - 让 3 个评委各自打分取平均（人工标注）
> - 同时让另一个 AI 裁判也打分（GPT-Eval）
> - 统计回复中"呀、嘛、哈"的数量（规则指标）
> - 最后取前 10 个，让真人客户盲选（AB Test）
>
> 这才是"定义和评测"。不是"我感觉"，而是"我测出来的"。

### Q9：口语化能力和理解能力有什么区别？

#### 面试官真正想问什么

面试官注意到候选人在混淆两个概念，这个问题在帮他**厘清边界**。

这也是面试官的"教学时刻"——当面试官开始帮你纠正概念时，说明他已经进入"考察你学习能力"的模式。此时你的反应非常关键：快速理解、举一反三、当场纠正自己的表述。

#### 严格区分

| | 理解能力 | 口语化能力 |
|------|---------|----------|
| **管什么** | 管"对不对" | 管"像不像人说的" |
| **出错表现** | 答非所问、幻觉 | 回答正确但像机器人 |
| **评测方式** | 准确率、召回率 | 自然度评分、偏好投票 |
| **根因** | 知识不够、推理不好 | 训练数据太正式、对齐过猛 |
| **优化方式** | 加知识、强化推理链 | 加口语化 SFT 数据、调整温度 |

#### 经典对比案例

```
用户：帮我看看明天北京天气怎么样

❌ 理解差 + 口语化好：
"好嘞！我帮你查查！嗯...明天...好像是，不对，
后天...哎呀我不太确定诶，要不你下个天气 App？"
→ 很会聊天，但没干活

❌ 理解好 + 口语化差：
"查询结果：2026年5月29日，北京市，天气晴，
气温18-28摄氏度，北风3-4级。查询完毕。"
→ 干活了，但听的人想打他

✅ 理解好 + 口语化好：
"明天北京大晴天，18 到 28 度，北风稍微有点大，
出门的话带件薄外套就够啦~"
→ 干活了，听着也舒服
```

#### 通俗故事：翻译官的故事

> 两个翻译官陪外国客户吃饭。
>
> **翻译官 A（理解差 + 口语化好）**：
> 客户说"I'm allergic to peanuts."
> A 翻译："这位先生说他心情不太好。"（理解错了）
> 但说得特别自然："哎呀这位先生今天可能有点心事~"
> → 客户吃了花生进医院了
>
> **翻译官 B（理解好 + 口语化差）**：
> 准确翻译了："这位先生对花生过敏。"
> 但是全程面无表情，说话像念说明书。
> → 客户没死，但觉得这顿饭吃得很尴尬
>
> **理想翻译官**：
> "哎对了，这位先生花生过敏，麻烦厨房注意一下哈！"
> → 既救了命，说话也好听
>
> **所以口语化和理解是两码事。一个保底（不出错），一个加分（体验好）。**

### Q10：训练时肯定要有一个指标吧？不可能全靠人工干预吧？

#### 面试官真正想问什么

"你做事情有没有闭环？"

训练模型不是玄学——你不能说"训完了，感觉还行"。面试官要看到你知道如何**用数据驱动的方式迭代模型**。

#### 指标体系搭建

```
训练阶段的指标金字塔：

          ┌─────────────┐
          │  业务指标    │  ← 最终目标（转化率、满意度）
          │  (AB Test)  │     上线后测量，1-2周出结果
          ├─────────────┤
          │  评测指标    │  ← 离线代理（GPT-Eval、人工标注）
          │  (Eval Set) │     训练完后跑，1小时出结果
          ├─────────────┤
          │  训练指标    │  ← 训练过程中监控（Loss、PPL）
          │  (Training) │     每个 step 都能看到
          └─────────────┘
```

**每一层的意义：**

- **训练指标（Loss/PPL）**：告诉你模型在"学习"。Loss 下降说明在学习。但 Loss 低不意味着模型好——可能过拟合了。
- **评测指标（Eval Set 准确率）**：告诉你模型在代表性样本上的表现。这是最常用的离线验证。
- **业务指标（AB Test 转化率）**：告诉你模型对真实业务有没有帮助。这才是终极答案。

**三者必须打穿**：训练指标好 → 评测指标好 → 业务指标好，缺一环都不行。

#### 通俗故事：健身的三面镜子

> 你健身减脂，有三面镜子：
>
> **第一面镜子（训练指标 = Loss）**：
> 你每天站在体重秤上。数字往下掉，说明方法有效。但体重下降可能是因为脱水，不一定是因为减脂。
>
> **第二面镜子（评测指标 = Eval Set）**：
> 你每两周在固定光线下、穿同一件衣服、摆同一个姿势拍对比照。这比体重秤更真实地反映体态变化。但不能天天拍，太麻烦。
>
> **第三面镜子（业务指标 = AB Test）**：
> 你每个月测一次 5 公里配速。这才是你健身的真正目的——跑得更快。体重降了但配速没变？那你的训练计划可能有问题。
>
> **最佳策略**：
> 每天上秤（Loss），两周拍一次照（Eval），一个月跑一次 5 公里（AB Test）。
> 三者如果方向一致（体重降 + 线条好 + 配速快），说明训练方法是正确的。
> 如果体重降了但配速没变，说明你可能在掉肌肉，得调整。

### Q11：你试过了吗？

#### 面试官真正想问什么

这个问题极其简短，但杀伤力极大。

面试官在问：**你刚才说的所有判断，是基于真实实验的，还是基于"我觉得"的？**

"试过了"→ 拿出数据 → 你说的是事实
"没试过"→ 你说的只是观点 → 前面所有对话可信度打对折

#### 强回答模板

> "试过了。我们把 7B、32B 和量化后的 32B 放在同一个评测集上跑了一遍：
>
> - 口语化评分：7B 2.8/5，32B-BF16 4.2/5，32B-INT4 4.0/5
> - 指令遵循：7B 68%，32B-BF16 89%，32B-INT4 87%
> - 推理延迟 P99：7B 120ms，32B-BF16 380ms，32B-INT4 180ms
>
> 结论很清晰：7B 效果不够，32B 量化到 INT4 后效果基本没降但速度翻倍。所以最终选了 32B-INT4。"

> 你说"这双鞋最合脚"。
>
> 面试官问："你试过了吗？"

> 你说"试过了，我试了 40、41、42 三双，41 的最合脚。40 的挤脚指头，42 的掉跟。"

>
> 你说"我觉得 41 应该最合脚吧"。

### Q12：怎么部署的？用什么架构？

#### 面试官真正想问什么

"你参与过真正的工程落地吗？还是只在笔记本上跑过 demo？"

部署是把模型变成产品的"最后一公里"，直接决定了用户能不能用上、用不用的爽。

#### 典型部署架构

```
                  ┌──────────────┐
                  │   客户端      │
                  │ (App/Web/API)│
                  └──────┬───────┘
                         │ HTTPS
                  ┌──────▼───────┐
                  │   API 网关    │  ← 鉴权、限流、路由
                  │  (Nginx/Kong)│
                  └──────┬───────┘
                         │
              ┌──────────┼──────────┐
              │          │          │
       ┌──────▼──┐ ┌────▼───┐ ┌───▼──────┐
       │ 模型副本1│ │模型副本2│ │ 模型副本3 │  ← 多副本负载均衡
       │ GPU 0   │ │ GPU 1  │ │ GPU 2    │
       │ vLLM    │ │ vLLM   │ │ vLLM     │
       └─────────┘ └────────┘ └──────────┘
                         │
                  ┌──────▼───────┐
                  │  监控告警     │  ← Prometheus + Grafana
                  │ (延迟/错误率) │
                  └──────────────┘
```

**各组件干什么：**
- **vLLM / TGI / TensorRT-LLM**：推理引擎，把模型的推理速度榨到最快
- **API 网关**：统一的入口，负责认证、限流、把请求分到空闲的 GPU
- **多副本**：一个 GPU 不够用？多复制几份，轮流接客
- **监控**：P99 延迟、错误率、GPU 利用率、显存使用——全盯着

#### 通俗故事：开奶茶店

> 训练出模型 = 你研发了一杯特别好喝的奶茶
>
> 部署 = 你开了一家店卖这杯奶茶
>
> - **vLLM（推理引擎）**：你的奶茶机。同样的配方，好机器出杯快，差机器半天出一杯。
> - **GPU**：奶茶机的功率。一台高功率机器同时能做 3 杯。
> - **多副本**：高峰期一台机器不够用，你买了 3 台奶茶机，3 个员工各操作一台。
> - **API 网关**：前台接单员。他收钱、写单子、看哪台机器空着就把单子派过去。
> - **负载均衡**：前台知道别把 10 个单子全给一个员工，轮流来。
> - **监控**：你在墙上挂了一个大屏幕，实时显示：每杯做了多久、有没有做错的、哪台机器快坏了。
>
> 面试官问"怎么部署的"，就是想听你说清楚这杯奶茶从下单到拿到手，中间经过了什么。

### Q13：QPS 多少？

#### 面试官真正想问什么

"你对性能有直觉吗？"

技术人对自己系统的 QPS 没有概念，就像司机不知道自己的车百公里几个油。

#### 什么是 QPS（每秒查询数）

```
QPS = 1 秒内能处理多少个用户请求

单卡 32B-INT4 典型 QPS 范围：
  - 短文本（20 token 输入，50 token 输出）：50-100 QPS
  - 中文本（200 token 输入，200 token 输出）：10-30 QPS
  - 长文本（2000 token 输入，500 token 输出）：2-5 QPS

影响因素：
  - 输入越长 → QPS 越低（上下文处理耗时）
  - 输出越长 → QPS 越低（生成耗时）
  - batch size 越大 → QPS 越高（但单条延迟也高）
  - 量化精度越高 → QPS 越低（计算量大）
  - GPU 越好 → QPS 越高
```

#### QPS 在你业务中意味着什么

```
假设你的对话服务：
  - 日活用户 10 万
  - 每人每天平均 10 次对话
  - 日均请求量 = 100 万次
  - 高峰时段（晚 8-10 点）占全天 40% = 40 万次
  - 高峰 QPS 需求 = 40 万 / 7200 秒 ≈ 55 QPS

如果单副本 QPS = 20，你需要至少 3 个副本（再加 20% 余量 → 4 个）
月成本 = 4 × GPU 单卡月租
```

#### 通俗故事：加油站

> QPS 就是你的加油站一秒钟能给几辆车加满油。
>
> 小型车（短文本）：加得快，一分钟能加 10 辆。
> 大型卡车（长文本）：油箱大，一分钟只能加 2 辆。
>
> 如果你的加油站在高速路口，高峰期每小时来 200 辆车（QPS 需求 = 200/3600 ≈ 每 18 秒一辆）。
>
> 你一个加油枪（单副本）每 30 秒加一辆车 → 不够。
> 装上 4 个加油枪（4 副本），每 30 秒可以同时加 4 辆 → 够了。
>
> 面试官问"QPS 多少"——他想知道你对"这个加油站能不能撑住高峰期"有没有概念。

### Q14：为什么量化到 4-bit？BF16、FP8 效果都比 4-bit 好，为什么不用？

#### 面试官真正想问什么

又一个 trade-off 问题。跟上文选 32B 的逻辑异曲同工——面试官想看你的决策过程是不是**一致且系统化**的。

#### 什么是量化

首先搞清楚量化的本质：**用更少的比特数存储模型的权重和激活值**。

```
BF16（16-bit）：每个数字用 16 个比特存储
  → 精度高、显存大、计算慢

FP8（8-bit）：每个数字用 8 个比特存储
  → 精度中等、显存减半、计算加速

INT4（4-bit）：每个数字用 4 个比特存储
  → 精度有损、显存减到 1/4、计算极快
```

**类比**：一张照片存成 RAW（BF16）、JPEG 高质量（FP8）、JPEG 低质量（INT4）。

#### 32B 模型的显存计算

```
32B 模型在不同精度下的显存需求（粗略计算）：

BF16：32B × 2 bytes = 64GB  → 需要 2 张 A100(80G) 或 1 张 H100(80G)
FP8：  32B × 1 byte  = 32GB  → 1 张 A100(40G) 能装下
INT4： 32B × 0.5 byte = 16GB  → 1 张 A10(24G) 就够了，还有余量
```

成本差异：
- 2 张 A100 月租：约 3-4 万
- 1 张 A100 月租：约 1.5-2 万
- 1 张 A10 月租：约 5000-8000

#### 为什么选 INT4

```
决策逻辑：

Step 1: 先确认效果是否可接受
  → 在评测集上对比 BF16/FP8/INT4 的效果
  → INT4 相比 BF16，口语化评分从 4.2 降到 4.0（降幅 5%，不可感知）
  → INT4 相比 BF16，指令遵循从 89% 降到 87%（降幅 2%）

Step 2: 再看成本和速度收益
  → INT4 显存只需 BF16 的 1/4 → GPU 成本降到 1/4
  → INT4 推理速度比 BF16 快约 2-3 倍
  → P99 延迟从 380ms 降到 180ms

Step 3: 做出决策
  → 花 1/4 的钱，获得快 2-3 倍的速度
  → 效果几乎感知不到差异
  → 结论：INT4 是最优解
```

#### 为什么不选 FP8 作为折中方案

FP8 确实是 BF16 和 INT4 之间的"甜点"，但它不是免费的：
- FP8 需要硬件支持（H100/H200 才有原生 FP8），A100 不支持
- 如果用 A100 跑 FP8，要软件模拟，速度反而更慢
- 如果硬件不支持 FP8，那 INT4 就是最好的选择

#### 通俗故事：压缩照片

> 你拍了一张 200MB 的超高清照片（BF16），要发到网上。
>
> **BF16（原图）**：200MB，完美画质。但上传要 2 分钟，你的云存储账单哗哗涨。而且用户用手机 6 寸屏幕看，根本看不出和压缩版的区别。
>
> **FP8（高质量压缩）**：50MB，画质几乎无损。上传快了很多。但你的电脑显卡太老，FP8 这个格式它不认识，要先转换成别的格式，转换过程反而更慢了。
>
> **INT4（低质量压缩）**：10MB，放大到 200% 能看出一点点噪点。但在 6 寸手机屏幕上？完全看不出来。上传秒传。存储费用降到 1/4。
>
> 你的用户 99% 用手机看，不放大到 200%。所以 INT4 看起来"差"，但对于实际使用场景来说是"够了"。
>
> 面试官追问："BF16 效果更好你为什么不用？"
> 你的回答："原图是好，但我的用户用手机看，一个月传 100 万张，流量账单是你付吗？"

### Q15：如果追求轻量化，为什么不用 7B？

#### 面试官真正想问什么

这个问题把「模型选型」和「量化选型」两个独立决策串起来了。

逻辑链：
- 你选 32B 是因为效果比 7B 好
- 但你把 32B 量化到 4-bit，效果下降了
- 那量化后的 32B 还比 7B 强吗？如果强不了多少，为什么不用更简单的小模型？

这是一个**非常好的追问**，因为它迫使你证明：量化后的 32B（INT4）仍然显著优于全精度 7B（BF16）。

#### 量化后的效果对比

```
关键洞察：量化损失 ≠ 参数量损失

量化（32B BF16 → 32B INT4）：
  - 每个参数的精度降低
  - 但 32B 个参数的知识都在
  - 效果损失：约 2-5%

小模型（32B BF16 → 7B BF16）：
  - 参数从 320 亿降到 70 亿
  - 知识容量大幅减少
  - 效果损失：约 20-30%
```

**结论：量化是"压缩"而非"裁剪"。** 32B INT4 仍然拥有 320 亿参数的知识广度，只是每个参数的表达精度略有下降。而 7B 从根本上就少了 250 亿参数。

#### 量化后对比实验

```
实测对比（在口语化对话评测集上）：

7B-BF16（全精度小模型）：
  - 口语化评分：2.8/5
  - 指令遵循：68%
  - 延迟 P99：120ms
  - 部署成本：低

32B-INT4（量化大模型）：
  - 口语化评分：4.0/5  ← 远好于 7B
  - 指令遵循：87%     ← 远好于 7B
  - 延迟 P99：180ms   ← 略慢但可接受
  - 部署成本：低       ← 相近
```

**故事就是：量化的 32B 在效果上仍然碾压全精度的 7B，且成本相近。**

#### 通俗故事：降薪的老员工 vs 新来的毕业生

> 你的公司预算紧张，有两个省钱方案：
>

### Q16：线上问题怎么发现的？只能客户反馈？

#### 面试官真正想问什么

"你有没有建立系统的可观测性？还是会出问题了才知道？"

这是区分"交付工程师"和"运维工程师"的关键。写好代码部署完 == 交付。部署完还能持续监控 == 运维。

#### 线上质量保障体系

```
被动发现（用户反馈）：
  └── 最慢、最差、最被动
  └── 用户投诉"你这个助手说话像机器人" ← 已经出问题了

主动监控（分三层）：

Layer 1: 服务健康监控（秒级）
  ├── 延迟（P50/P99）：有没有突然变慢？
  ├── 错误率：5xx 比例有没有突增？
  ├── QPS：有没有异常波动？
  └── GPU 显存/利用率：有没有 OOM？

Layer 2: 输出质量监控（分钟级）
  ├── 空回复率：模型是不是频繁输出空内容？
  ├── 重复率：是不是陷入了重复循环？
  ├── 乱码率：是不是偶尔吐出乱码？
  └── 输出长度异常：突然回复变短/变长？

Layer 3: 业务指标监控（小时/天级）
  ├── 用户满意度评分有没有下降？
  ├── 对话完成率有没有变化？
  ├── 次日留存有没有波动？
  └── 关键转化指标有没有异常？
```

#### 告警 + 响应流程

```
监控触发 → 告警通知 → 初步排查 → 回滚/止损 → 根因分析 → 修复上线

每一步都要有预案：
  - 告警通知谁？（值班人、开发、PM）
  - 回滚到哪个版本？（上一个稳定版本必须可一键回滚）
  - 止损手段？（切换备份模型、降级到规则兜底）
```

#### 通俗故事：饭店的卫生监控

> 你开了一家饭店，怎么保证食物不出问题？
>
> **纯靠客户反馈（被动）**：
> 客人吃了拉肚子，打 12315 投诉，你才知道。这时候已经被查封了。
>
> **主动监控（主动）**：
> - **后厨监控（服务健康）**：随时看厨师有没有按规定戴手套、食材有没有放冰箱。冰箱温度一高就自动报警。
> - **随机抽样（输出质量）**：每天中午大厨从出餐口随机端走三盘菜尝尝——不用等客人说难吃，你自己先发现。
> - **经营数据（业务指标）**：每周看回头客比例。如果上周 40% 这周 30%，说明菜可能出了问题，主动排查。
>
> 线上模型也一样——等你收到用户投诉，差评可能已经满天飞了。主动监控的意思是：**在用户发现之前，你已经知道了。**

### Q17："难负样本"是什么意思？

#### 面试官真正想问什么

"你真的做过微调吗？还是照着手册抄的？"

难负样本（Hard Negative Samples）是微调中一个高级概念——没有实际做过数据工程的人不会注意到这个概念。

#### 什么是难负样本

在微调中：
- **正样本**：告诉模型"应该这样回答"
- **负样本**：告诉模型"不能这样回答"
- **难负样本**：模型实际会犯错、恰好踩在"边界"上的差回答

```
普通负样本（已经避免了）：
  用户："今天天气怎么样"
  ❌ 模型："我喜欢吃香蕉"  ← 这种错误模型早就不犯了
  → 加这种负样本是无效训练

难负样本（实际会犯的）：
  用户："今天天气怎么样"
  ❌ 模型："根据气象部门发布的最新数据，今日气温..."
  → 内容没错，但太正式了，不够口语化
  → 这种"对的但不好的"才是难负样本
```

#### 难负样本的来源

1. **线上 Bad Case**：用户点赞/点踩的数据 → 被点踩的就是难负样本
2. **对比实验**：AB Test 中对照组输了的原因 → 分析输在哪
3. **边界制造**：故意设计容易混淆的 Prompt → 诱导模型犯错
4. **人工构造**：让标注员专门找"勉强能接受但不够好"的回复

#### 为什么难负样本重要

想象模型是一个边界越来越清晰的分类器。普通负样本离边界很远（模型永远不可能犯），加再多也没用。难负样本刚好在边界上，每一个都能重新刻画边界。

```
模型的能力边界：
        好 ┃ 差
           ┃
    ✅✅✅ ┃ ❌❌❌  ← 边界
    ✅✅✅┃❌❌❌
    ✅✅ ┃ ❌❌❌
      普通 ┃ 普通
      正样 ┃ 负样
      本   ┃ 本
           ┃
    ✅ ┊ ❌  ← 难负样本！刚好在边界上的
    ✅✅┊❌     这些样本重新刻画边界的位置
```

#### 通俗故事：驾校教练

> 驾校教练教倒车入库：
>
> **正样本**："看，方向盘打到这里，看后视镜这个点，慢慢倒，完美入库。"
>
> **普通负样本**："别把车开到沟里去。"——你本来就不会开到沟里。说一万次也没用。
>
> **难负样本**："你每次回方向的时机都晚了 0.5 秒。记住了：保险杠到黄线三分之二的时候就要回方向，不是全到了才回。你看，差这 0.5 秒，车尾就偏了 15 厘米。考官扣分就扣在这里。"
>
> 教练精准击中了你**实际上会犯但自己意识不到**的错误。这就是难负样本。
>
> 微调模型也是一样——用客户真实点踩的回复做训练数据，远比用"这个模型不会犯的错误"做负样本有效。

### Q18：微调了几轮？

#### 面试官真正想问什么

"你的迭代节奏合理吗？"

一次就成功 → 说明任务太简单或者你没发现隐藏问题。
太多轮还不收敛 → 说明方法有问题。

#### 典型迭代节奏

```
Round 1：基线微调
  - 用初始 SFT 数据（正样本 + 普通负样本）
  - 目标是让模型基本理解任务格式
  - 预期：loss 收敛，但效果一般

Round 2：难负样本挖掘
  - 分析 Round 1 的 Bad Case
  - 补充难负样本 + 针对性正样本
  - 预期：在 Bad Case 类型上有明显提升

Round 3：边界打磨
  - 分析 Round 2 残留的边界问题
  - 小批量高质量数据做精细调整
  - 预期：整体效果稳定，Bad Case 率降到目标线以下

通常 3-5 轮是比较合理的范围
```

#### 通俗故事：写作文

> 老师布置了一篇作文：
>
> **第一稿**：你写完了，通顺但不出彩。老师批注：开头太平淡，结尾没有升华。
>
> **第二稿**：改了开头和结尾，好了很多。但老师又指出：中间有一段跟主题没关系，删掉。
>
> **第三稿**：删掉废话后逻辑更紧凑了。老师说：很好，只剩几个不自然的用词，微调一下。
>
> **定稿**：从 80 分改到了 95 分。
>
> 三稿是合理的。如果你说"一遍就写完了，老师没改"，只有两种可能：你是天才（极少见），或者老师根本没仔细看。
>
> 微调同理——一轮就收敛只有两种解释：你的评测不够严格，或者你已经在这个任务上做了很久有积累。

### Q19：T0 转换率是什么？注册转换率是什么？+4 个点怎么来的？

#### 面试官真正想问什么

"你懂得怎么把技术指标翻译成业务价值吗？"

这个问题的本质是：**你会不会向老板/产品经理讲明白你的工作值多少钱。**

#### 两个指标的定义

**T0 转换率（首次对话转化率）**：
```
T0 转换率 = 完成目标动作的用户数 / 首次发起对话的用户数

场景：用户第一次跟你的 AI 助手聊天
目标：让用户完成某个动作（下单、注册、预约等）

举例：
  - 100 个用户首次跟 AI 客服聊天
  - 其中 15 个最终下了单
  - T0 转换率 = 15%
```

**注册转换率**：
```
注册转换率 = 完成注册的用户数 / 进入注册流程的用户数

举例：
  - 200 个用户被引导到注册页面
  - 其中 40 个完成了注册
  - 注册转换率 = 20%
```

#### +4 个点是怎么来的

```
AB Test 设置：
  - 对照组（50% 流量）：使用旧版模型（7B-BF16）
  - 实验组（50% 流量）：使用新版模型（32B-INT4）

运行 2 周后：

对照组：
  - 首次对话用户：50,000
  - 完成目标用户：6,000
  - T0 转换率：12%

实验组：
  - 首次对话用户：50,000
  - 完成目标用户：8,000
  - T0 转换率：16%

差异：+4 个百分点（相对提升 33%）
统计显著性：p < 0.01（置信度 99%+）
结论：口语化模型显著提升了用户转化
```

#### 如何向老板讲这个故事

```
"老板，我们花 1.5 万/月部署了新的口语化模型。
AB Test 跑了 2 周，T0 转换率从 12% 提升到 16%。

换算成业务价值：
  - 每月首次对话用户：10 万
  - 多转化的用户：10万 × 4% = 4000 人
  - 每个转化用户价值：200 元
  - 月增收入：4000 × 200 = 80 万

投入：1.5 万/月
产出：80 万/月
ROI：53 倍
"
```

这才是面试官想听到的回答——**不是只报数字，而是能从技术讲到业务价值。**

#### 通俗故事：超市导购

> 你是一家超市的经理，想提升销售额。
>
> **旧方案（7B 通用回复）**：门口站一个保安，顾客问"洗衣液在哪"，保安手一指："A 区 3 排。"顾客自己去找了。100 个问路的，最后买洗衣液的只有 12 个。
>
> **新方案（32B 口语化回复）**：门口站一个热心大姐。"洗衣液啊？我带你去！哎对了你家有几口人啊？人多的话买这个家庭装划算，现在正好买二送一。"100 个问路的，16 个买了——多了 4 个。
>
> 4 个点的差异就是这个热心大姐多卖出去的。你给大姐的工资是一天 200 块，但她帮你多卖了 1600 块的洗衣液。
>
> **面试官问"T0 转换率是什么"——你在讲的不只是模型好坏，而是"换了热心大姐后多赚了多少钱"。**

### Q20：难负样本对我进行微调，大概是什么意思？

#### 面试官真正想问什么

面试官在确认你真的理解"难负样本→微调"这个完整链路，而不是只是背定义。

#### 完整流程

```
Step 1: 采集难负样本
  └── 来源：线上点踩、AB Test 对比输掉的回复、人工质检标记

Step 2: 构建训练数据
  └── 格式：(用户问题, 差回复[难负样本], 好回复[正样本])
      训练目标是让模型学会：遇到这个问题 → 不要给出这个差回复 → 要给出这个好回复

Step 3: 微调（Fine-tuning）
  └── 用 LoRA/QLoRA 在难负样本 + 正样本混合数据集上微调
      学习率比第一轮更低（防止破坏已有能力）
      训练 epoch 更少（1-3 epoch，防止过拟合）

Step 4: 验证
  └── 在留出的难负样本测试集上验证
      关键指标：模型在新难负样本上的"犯错率"下降了多少
```

#### 通俗故事：错题本

> 微调模型就像帮学生提分。
>
> **普通方法**：把课本从头到尾抄一遍。花时间多，效果差，因为 90% 的内容学生本来就会。
>
> **难负样本方法**：把学生所有做错的题整理成"错题本"，针对每道错题：
> - 左边：学生写的错误答案（难负样本）
> - 右边：正确的解题步骤（正样本）
>
> 不用抄整本课本，就盯着错题反复练。10 道错题的训练效果远大于 100 道他已经会的题。
>
> 这就是"基于难负样本做微调"——用最少的数据，打最准的补丁。

### Q21：LandGauge 和 LandTrain 有了解吗？跟 IG 什么关系？LandGauge 具体怎么实现？

#### 面试官真正想问什么

有三个层次：
1. 你是否关注业界前沿工具？（广度）
2. 你是否理解工具之间的关系？（系统思维）
3. 你是否了解工具的内部原理？（深度）

#### LandGauge 是什么

Meta 推出的大模型评估框架，核心思路是**标准化、多维度、可复现的模型评测**。

核心能力：
- 预置评测维度：事实准确性、安全合规、推理能力、指令遵循等
- 标准化评测集：每个维度有配套的高质量测试数据
- 自动化评分：用规则 + LLM-as-Judge 自动打分
- 可视化报告：生成分维度雷达图，一目了然

#### LandTrain 是什么

与 LandGauge 配套的训练优化工具，形成"评测→训练→再评测"的闭环。

```
LandGauge 评测 → 发现短板 → LandTrain 针对性训练 → LandGauge 再评测
```

#### IG（Integrated Gradients，积分梯度）

一种模型可解释性方法，回答"模型为什么给出了这个输出"。

核心原理：
```
1. 给定输入（比如一句话）
2. 从"空白输入"逐步插值到"实际输入"
3. 计算每一步的梯度
4. 沿路径积分，得到每个输入 Token 对输出的贡献分数
5. 分数越高的 Token，对模型决策影响越大
```

#### 三者的关系

```
LandGauge：告诉你模型得了多少分（诊断）
IG：告诉你为什么得这个分数（病因分析）
LandTrain：告诉你该怎么提高分数（治疗方案）

三者形成闭环：
LandGauge 说"口语化评分低"
  → IG 分析发现"模型对用户口语词（'嘛'、'呀'）的注意力太低"
  → LandTrain 针对性地加口语词训练数据
  → LandGauge 再测"口语化评分提升了"
```

#### 通俗故事：体检 + 化验 + 康复训练

> **LandGauge = 体检中心**
> 你去做了一个全面体检，拿到报告：
> - 血糖：偏高
> - 肺活量：偏低
> - 体脂率：超标
> - 骨密度：正常
>
> 你知道自己哪里有问题了。
>
> **IG = 化验单**
> 体检说血糖高，但不知道为什么。化验单告诉你：
> "过去三个月你的碳水摄入超标 40%，这是血糖偏高的主要原因。脂肪和蛋白质摄入正常。"
> 现在你知道病因了。
>
> **LandTrain = 康复训练计划**
> 体检（LandGauge）说血糖高，化验单（IG）说碳水吃太多，康复师（LandTrain）给你开处方：
> "早餐的白米饭换成燕麦，午餐的米饭减半，晚上不吃主食。每天快走 30 分钟。一个月后复查。"
>
> 一个月后再去体检 → 血糖降了 → 闭环完成。
>
> **三个工具缺一不可。只体检不治疗（只用 LandGauge）是耍流氓。不体检就瞎练（不用 LandGauge）是找死。**

### 面试官的三种追问模式

| 模式 | 例子 | 目的 |
|------|------|------|
| **反问** | "7B 不应该更好吗？" | 测试你是否动摇、是否真正理解 |
| **对比追问** | "BF16 更好为什么不用？" | 测试你是否清楚 trade-off |
| **拆概念** | "口语化和理解有什么区别？" | 测试你是否理解概念边界 |

### 什么是好回答

好的回答不是"最正确的回答"，而是**展示出你有一套可迁移的思考框架**。

面试官不在乎你选的是 32B 还是 33B，他在乎的是：**下次遇到不一样的任务，你还能用同样的方法选出合适的方案吗？**

### 三条黄金法则

1. **量化一切**：不要说"还行"，说"评测集上准确率从 68% 提升到 87%"
2. **证明你试过**：不要说"应该是"，说"我们实测过，数据是..."
3. **区分概念**：当面试官帮你区分概念时（比如口语化 vs 理解）——立刻接住、举一反三、当场纠正自己的表述

---

> **最后讲一个故事。**
>
> 有个木匠去面试。面试官问他："为什么这把椅子用橡木不用松木？"
>
> 差的回答："橡木硬。"
>
> 好的回答："橡木硬（硬度 600 vs 300），适合做椅子腿，不会坐几年就摇晃。而且我们试了松木，用了一年开裂了。"
>
> 最好的回答："我们定了一个选材标准：椅子腿承重 > 200kg，椅子面触感不能太粗糙，成本控制在 200 块以内。橡木腿 180 块，松木腿 80 块但承重只有 150kg，黑胡桃 300 块超预算。橡木是唯一满足三个底线的材料。"
>
> 面试官录用了第三个，不是因为他说橡木好——**而是因为他有一套任何时候都能选出"最合适材料"的方法。**
